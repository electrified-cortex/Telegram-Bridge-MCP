## Executive Summary

This release adds a new per-session activity-file subsystem and relies on it to change dequeue behavior. The main risk is not the debounce math itself; it is that the bridge now exposes a host filesystem write primitive to any caller holding a valid session token, and the implementation treats failed touches as if they succeeded. That combination creates both a security expansion and a silent-correctness failure in the core wakeup path. Overall risk is **moderate to high**: no obvious remote code execution bug in the diff alone, but one substantial blast-radius increase and multiple ways for the wakeup signal to fail silently under real operating conditions.

## Findings

### Critical

No critical findings from the diff alone.

### High

#### H1 — Arbitrary host-file append primitive behind a session token

- **File:** `src/tools/activity/file-state.ts` (approx. lines 25-37, 40-50), `src/tools/activity/create.ts`, `src/tools/activity/edit.ts`
- **Severity:** High
- **Description:** `file_path` accepts any absolute path after only three checks: non-empty, no null byte, no `..`, and `isAbsolute(...)`. Once registered, `appendNewline()` appends `"\n"` to that path whenever activity is touched.
- **Why real:** This turns a valid session token into an arbitrary append-to-file capability with the bridge process's OS permissions. Append-only is still dangerous: it can corrupt config files, mutate shell/profile files, poison logs, trigger filesystem watchers, or write to Windows UNC paths if `isAbsolute()` accepts them. The `..` check does not mitigate the real problem because the implementation already allows any absolute destination.

#### H2 — Failed touches recorded as successful; wakeups can be black-holed silently

- **File:** `src/tools/activity/file-state.ts` (approx. lines 40-56, 90-118)
- **Severity:** High
- **Description:** `doTouch()` sets `lastTouchAt = Date.now()` before the async append actually completes, and `appendNewline()` swallows errors into `console.warn()`.
- **Why real:** A missing parent directory, permission error, Windows sharing violation, or transient filesystem failure still advances debounce/suppression state as if the file had been touched. After that, `touchActivityFile()` can suppress further attempts for up to `MAX_INTERVAL_MS` or until later activity arrives. The bridge can report a registered activity file while silently failing to emit the wakeup signal the agent depends on.

### Medium

#### M1 — Activity suppression can drop the only wakeup edge; dequeue cap only compensates one call pattern

- **File:** `src/tools/activity/file-state.ts` (approx. lines 97-118), `src/tools/dequeue.ts`
- **Severity:** Medium
- **Description:** In `touchActivityFile()`, when `agentRecentlyActive` is true and no debounce timer exists, the function returns without touching the file and without scheduling a trailing-edge touch. The dequeue safeguard only caps timeouts when `timeout === undefined`.
- **Why real:** The activity-file mechanism is no longer self-sufficient. If a caller uses an explicit long dequeue timeout, or any consumer relies on the file touch as the primary wakeup signal, the first queued message after "recent activity" can produce no file change at all. The message will then sit until the explicit timeout expires or a later unsuppressed event fires.

#### M2 — Delete and rotation paths claim success when the owned file is still on disk

- **File:** `src/tools/activity/file-state.ts` (approx. lines 69-80), `src/tools/activity/delete.ts`, `src/tools/activity/edit.ts`
- **Severity:** Medium
- **Description:** `clearActivityFile()` swallows all `unlink()` failures for TMCP-owned files. `handleActivityFileDelete()` reports `deleted_file: wasOwned` regardless of whether deletion actually happened.
- **Why real:** On Windows especially, deleting an open or watched file often fails. Create/edit flows will rotate to a new file while leaving the old one behind. Operators are told cleanup succeeded while orphaned activity files and stale wakeup artifacts accumulate.

### Low

No low-severity findings. Meaningful issues are higher impact than style or polish.

## Architectural Concerns

1. **Feature coupling:** The new design couples two independent mechanisms — filesystem wakeups and dequeue timeout behavior. Correctness now depends on both being used together in a specific way, which is brittle and easy for future callers to violate.

2. **Overly broad file access:** External activity files should probably be capability-scoped, not "any absolute path." If arbitrary external paths are required, the feature needs a much stronger trust model and explicit blast-radius documentation.

3. **Best-effort cleanup only:** TMCP-owned files are cleaned up in-memory only with no startup reconciliation. If these files are part of a durable operator-facing contract, the system needs honest deletion status and orphan recovery rather than silent best-effort.

## Cross-Model Dispatch Pattern

This report was produced by dispatching GPT-5.4 via the `runSubagent` tool available in GitHub Copilot Worker agents.

**How it works:**
1. Worker (Copilot, Claude Sonnet 4.6) reads the git diff and prepares a structured prompt
2. `runSubagent(model: "GPT-5.4 (copilot)", prompt: ...)` dispatches a stateless subagent running on GPT-5.4
3. The subagent receives the full diff context + adversarial reviewer instructions
4. Result is returned synchronously to the Worker, which then files the report

**API surface used:** `runSubagent` tool (VS Code Copilot agent tool, `model` parameter)

**Available models (confirmed at time of dispatch):**
- `Claude Sonnet 4.6 (copilot)` — default Worker model
- `GPT-5.3-Codex (copilot)`
- `GPT-5.4 mini (copilot)`
- `GPT-5.4 (copilot)`
- `Claude Haiku 4.5 (copilot)`
- `Auto (copilot)`

**Latency/cost:** Synchronous; no streaming. No separate API key required — uses Copilot subscription. Suitable for one-off review tasks where a different model's perspective is valuable.

**Limitation:** Stateless subagent only. Cannot access workspace files directly — the caller must pass all context as prompt text. For large codebases, caller is responsible for diff/context scoping.
