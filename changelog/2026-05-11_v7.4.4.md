# v7.4.4 — 2026-05-11

## Fixed

- **Activity kick is now queue-gated** (10-0896): `doTouch` calls in both
  `touchActivityFile` (debounce-expiry path) and `handleSessionStopped` (stop-event path)
  are now conditional on `hasPendingUserContent(sid)`. If no messages are pending, the
  activity file is not touched. This eliminates the infinite self-kick loop in
  single-agent (Curator-pattern) setups where a Stop hook previously caused
  file-mtime → Monitor-wake → turn-end → Stop hook → repeat.

- **`handleSessionStopped` resets `lastActivityAt`** (10-0896, F1 nit from v7.4.3 review):
  `lastActivityAt` is now zeroed on stop, ensuring the next inbound message after a session
  stop triggers an immediate kick (no 60 s debounce wait).

### Subsumed from v7.4.3 code review

- F1 nit: `handleSessionStopped` leaving `lastActivityAt` stale — fixed here as part of
  the state-model alignment.

### State model alignment

Activity state is now explicitly modelled as `active` or `inactive`. A session is
`active` while any of: inflight dequeue, blocking interactive call, or a tool call
within the debounce window. It transitions to `inactive` on stop hook or debounce
expiry. Kicks only fire on transition to `inactive` AND only when the session queue
has pending messages. The debounce window governs when "inactive" is declared; it is
no longer a force-kick timer.
